Training Deck

Network Debugging With Docker Compose

A hands-on lab for interfaces, layer 2 neighbors, routes, gateway behavior, DNS, firewall policy, and packet capture.

`arm_net` = `172.21.0.0/24` | `brain_net` = `172.20.0.0/24`
←→Space
Objective

What The Team Should Learn

  • How to identify interfaces, subnets, and local neighbors.
  • How routing and a dedicated gateway move traffic across subnets.
  • How DNS on the router supports name-based troubleshooting.
  • How firewall policy changes protocol behavior without breaking everything.
  • How to debug systematically with `ip`, `ss`, `ping`, `tcpdump`, `arp-scan`, and `iptables`.

Topology

Two Subnets, One Router

  • `left_arm` and `right_arm` share the same layer-2 segment on `arm_net`.
  • `brain` is isolated on `brain_net` and only reachable through the router.
  • The router runs `dnsmasq`, forwards IPv4, and enforces `FORWARD DROP` plus explicit TCP/9000 allow rules.
  • Cross-subnet TCP works. Cross-subnet ICMP does not.
The router is the teaching gateway for inter-subnet traffic.
3 / 10
Layer 2

First Question: Who Is On My Local Segment?

docker compose exec left_arm bash
ifaces
ip neigh show dev eth0
arping -I eth0 right_arm
l2-neighbors

What To Emphasize

  • `right_arm` and `router` are L2 neighbors on `arm_net`.
  • `brain` is not an L2 neighbor from `left_arm`.
  • `ip neigh` shows the current ARP cache.
  • `arp-scan --interface=eth0 --localnet` actively sweeps the broadcast domain.

Also point out that Docker’s bridge endpoint at `.1` is visible at layer 2 even though your training gateway is `.254`.

Start local before blaming routes, DNS, or firewall policy.
4 / 10
Routing

Next Question: Where Will The Packet Go?

docker compose exec left_arm bash
routes
ip route get 172.20.0.11

docker compose exec brain bash
routes
ip route get 172.21.0.11

Expected Routing Logic

`left_arm` / `right_arm`

`172.20.0.0/24 via 172.21.0.254`

`brain`

`172.21.0.0/24 via 172.20.0.254`

This is the clean handoff between layer 2 and layer 3: `brain` is not on the local segment, so the packet goes to the router.

Route lookup answers “which next hop?” before you test the actual application.
5 / 10
DNS + Reachability

Names, Then Packets

docker compose exec left_arm bash
dns
nslookup brain
check-router
check-brain

Arm-side DNS

`172.21.0.254`

Brain-side DNS

`172.20.0.254`

Training Angle

  • Show that name resolution is handled by the router, not by Docker’s default embedded DNS.
  • Make the team distinguish “can I resolve the name?” from “can I complete the TCP flow?”
  • `check-brain` intentionally combines resolution, route inspection, and a TCP probe to keep the story coherent.
DNS success does not guarantee transport success.
6 / 10
Firewall

Allowed

Cross-subnet TCP to port `9000`

Observed

Name resolution and routing both succeed

Blocked

Cross-subnet ICMP such as `ping brain`

Show The Policy

docker compose exec router bash
demo-reset
fw
iptables -S FORWARD

Story To Tell

  • Default `FORWARD` policy is `DROP`.
  • Only explicit TCP/9000 rules permit cross-subnet forwarding.
  • Same-subnet traffic like `left_arm -> right_arm` never traverses the router.
  • Reset counters before the demo so the packet counts move in front of the audience.
One of the best teaching moments: same route, same DNS, different protocol result.
7 / 10
Packet Capture

Let The Audience See The Traffic

docker compose exec router bash
sniff-9000

docker compose exec left_arm bash
sniff-brain

printf "hello\n" | nc brain 9000

What Packet Capture Proves

  • TCP flows to `brain:9000` actually traverse the router.
  • Layer-2 neighbor discovery stays on the local segment.
  • When `ping brain` fails, the absence of replies is visible as well.

`sniff-brain` is more focused on a node. `sniff-9000` is better on the router when you want to show the path across subnets.

If the audience doubts the explanation, show them the packets.

Live Sequence

Recommended Demo Script

  1. `demo-reset` on `router`
  2. `check-self` on `left_arm`
  3. `l2-neighbors` on `left_arm`
  4. `check-router` on `left_arm`
  5. `ping right_arm`
  6. `ping brain`
  7. `check-brain`
  8. `sniff-9000` on `router` while repeating the TCP probe
  9. `iptables -vnL FORWARD` on `router` to close the loop

Why This Order Works

  • You move from local facts to routed facts.
  • You separate layer 2, layer 3, DNS, transport, and policy.
  • You show both working and failing paths without changing the topology mid-demo.
Keep the narrative linear: local segment → route → name → transport → policy.
9 / 10
Takeaways

What To Remember

  • Layer-2 neighbors answer “who is on my segment?”
  • Routes answer “where does the packet go next?”
  • DNS answers “what IP should I try?”
  • Firewall policy answers “which traffic is actually allowed through?”
  • A strong debugging habit is to verify each layer separately and only then connect the full story.
