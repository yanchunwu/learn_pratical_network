Network Training Lab

Print-oriented notes for a Docker Compose networking session covering interfaces, layer 2 neighbors, routes, gateway behavior, DNS, firewall policy, and packet capture.

Learning Goals

  • Recognize what is local to a subnet versus what must traverse a router.
  • Use the right tool for each layer: `ip neigh`, `arp-scan`, `ip route`, `nslookup`, `ss`, `iptables`, `tcpdump`.
  • Explain why TCP can work while ICMP fails in the same topology.

Key Design Choices

  • `left_arm` and `right_arm` are on the same L2 segment.
  • `brain` is on a different subnet, so it is reached through a dedicated router.
  • `dnsmasq` runs on the router so all three hosts resolve by name across subnets.
  • The router allows new cross-subnet TCP to port `9000` but drops other forwarded traffic such as ICMP.

Teaching Note

Docker still has its own bridge endpoint on each subnet at `.1`. In this lab, `.254` is the teaching gateway because the nodes install explicit routes to the router.

Layer 2

Who Is On My Local Segment?

Commands

docker compose exec left_arm bash
ifaces
ip neigh show dev eth0
arping -I eth0 right_arm
arping -I eth0 172.21.0.254
l2-neighbors

Interpretation

  • `right_arm` and `router` are layer-2 neighbors on `arm_net`.
  • `brain` is not a layer-2 neighbor because it is on `172.20.0.0/24`.
  • `ip neigh` shows what the kernel currently knows.
  • `arp-scan --interface=eth0 --localnet` actively discovers hosts on the broadcast domain.

Expected Neighbors

`172.21.0.12` (`right_arm`), `172.21.0.254` (`router`), and Docker’s bridge endpoint `172.21.0.1`.

Resolve local adjacency before looking at routes or firewall state.

Layer 3

Where Will The Packet Go?

`left_arm` / `right_arm`

routes
ip route get 172.20.0.11

Expected route: `172.20.0.0/24 via 172.21.0.254`

`brain`

routes
ip route get 172.21.0.11

Expected route: `172.21.0.0/24 via 172.20.0.254`

Teaching Point

Layer 2 tells you whether the destination is local. The route table tells you which next hop to use when it is not.

Route lookup is the bridge between local neighbor knowledge and end-to-end transport.

DNS + Transport

Can I Resolve The Name And Open The TCP Session?

Commands

docker compose exec left_arm bash
dns
nslookup brain
check-router
check-brain

DNS Servers

Arm side uses `172.21.0.254`. Brain side uses `172.20.0.254`.

Interpretation

  • Successful DNS means the router can answer the name.
  • Successful TCP means the route, gateway, DNS, and firewall policy all line up for port `9000`.
  • Keep the audience aware that name resolution and transport success are separate checks.
DNS answers “what IP?”; TCP answers “can I actually talk to it?”

Policy

Why TCP Works While ICMP Fails

Allowed

Cross-subnet TCP to port `9000`

Visible

Same routes and same DNS still apply

Blocked

Cross-subnet ICMP such as `ping brain`

Commands

docker compose exec router bash
demo-reset
iptables -vnL FORWARD
iptables -S FORWARD

Interpretation

  • Default `FORWARD` policy is `DROP`.
  • Explicit rules allow TCP/9000 between `eth0` and `eth1`.
  • Reset counters before the demo so the packet counts move during the session.
One topology, two protocols, two different outcomes because of policy.

Packets

Show The Traffic

Packet Capture

docker compose exec router bash
sniff-9000

docker compose exec left_arm bash
sniff-brain

printf "hello\n" | nc brain 9000

Suggested Narrative

  • Start capture on the router.
  • Generate a TCP probe from `left_arm`.
  • Show the audience that successful cross-subnet TCP visibly traverses the router.
  • Repeat with `ping brain` to show the contrast.
If the explanation is challenged, packet capture is the tie-breaker.

Presenter Notes

Recommended Live Sequence

  1. `demo-reset` on `router`
  2. `check-self` on `left_arm`
  3. `l2-neighbors` on `left_arm`
  4. `check-router` on `left_arm`
  5. `ping right_arm`
  6. `ping brain`
  7. `check-brain`
  8. `sniff-9000` on `router` while repeating the TCP probe
  9. `iptables -vnL FORWARD` on `router`

Closing Message

  • Local neighbor discovery is not the same as routed reachability.
  • DNS success is not the same as transport success.
  • Routes explain the path; firewall policy explains the outcome.
  • Good debugging isolates each layer before blaming the whole network.
